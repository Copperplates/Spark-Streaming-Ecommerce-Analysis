# 成员3（程觉晓）：实时处理与流式分析

## 一、简要概括

我主要负责实时处理模块，构建并验证了流式数据输入与 Structured Streaming 分析链路，使系统具备了对最新交易事件进行动态监测和分析的能力。

---

## 二、模块总体架构

实时处理链路由三个核心脚本协同构成一条完整的数据流水线：

| 脚本 | 职责 | 输入 | 输出 |
|------|------|------|------|
| `apps/realtime_producer.py` | 事件生成器（Event Producer） | 原始 CSV 种子数据 | JSON 事件文件 / Kafka 消息 |
| `apps/realtime_processor.py` | 流处理引擎（Stream Processor） | JSON 事件流（文件目录或 Kafka Topic） | 7 类结构化 CSV + HTML 仪表盘 |
| `apps/realtime_reporting.py` | 报表生成器（Report Renderer） | 流处理产出的 CSV 文件 | `dashboard.html` + `summary.md` |

三者关系如下：

```
realtime_producer.py ──生成事件──▶ realtime_input/ ──Spark监听──▶ realtime_processor.py
                                                                        │
                                                          ┌─────────────┘
                                                          ▼
                                                    Reports/realtime/
                                                  (CSV + HTML + MD)
                                                          ▲
                              realtime_reporting.py ──────┘
```

系统同时支持**文件流模式**（教学演示默认）和**Kafka 模式**（可选生产级模式），通过 `--mode` / `--source-type` 参数切换。

## 三、事件生成器（realtime_producer.py）详细设计

### 3.1 设计目标

批处理分析面向的是静态历史数据，而实时处理需要模拟"电商平台不断产生新交易"的场景。因此 `realtime_producer.py` 的设计不是随机凭空捏造数据，而是**以原始数据集为种子，通过受控变异产生逼真的交易事件**，使分析结果既保留原始数据的统计特征，又能体现动态变化。

### 3.2 核心实现

**种子数据加载**（`apps/realtime_producer.py:79-87`）：

```python
def load_seed_rows(path):
    with open(path, "r", encoding="utf-8-sig", newline="") as handle:
        reader = csv.DictReader(handle)
        rows = list(reader)
    if not rows:
        raise ValueError(f"No rows found in source file: {path}")
    return rows
```

从挂载到容器内的 `E-commerceCustomerBehavior-Sheet1.csv` 加载全部原始记录作为变异种子池。

**数据变异逻辑**（`apps/realtime_producer.py:98-133`）：

`mutate_row()` 函数对每条种子记录施加受控随机扰动，关键变异规则如下：

| 字段 | 变异方式 | 约束 |
|------|----------|------|
| `total_spend` | 原值 + Uniform(-90, +140) | 下限 120.0 |
| `items_purchased` | 原值 + Uniform(-2, +3) | 下限 1 |
| `average_rating` | 原值 + Uniform(-0.4, +0.5) | 区间 [2.8, 5.0] |
| `days_since_last_purchase` | 原值 + Uniform(-6, +8) | 下限 1 |
| `discount_applied` | 25% 概率翻转布尔值 | — |
| `age` | 原值 + Uniform(-1, +1) | 区间 [18, 70] |

每条事件被赋予唯一标识 `event_id`（格式 `evt-000001`）和时间戳 `event_time`（ISO 8601 UTC 格式），并基于变异后的 `average_rating` 和 `days_since_last_purchase` 动态计算满意度等级：

```python
def derive_satisfaction(average_rating, days_since_last_purchase):
    if average_rating >= 4.3 and days_since_last_purchase <= 28:
        return "Satisfied"
    if average_rating < 3.7 or days_since_last_purchase > 38:
        return "Unsatisfied"
    return "Neutral"
```

这一设计保证了每条事件都是完整且自洽的：满意度等级与评分、最近购买天数之间存在逻辑一致的推导关系。

**批量发布机制**（`apps/realtime_producer.py:157-185`）：

- 默认每批次生成 20 条事件（`--batch-size`），共发布 5 批次（`--iterations`），批次间隔 5 秒（`--sleep-seconds`）
- 文件模式下，每批写入一个 JSON Lines 文件 `batch_001.json` 到 `realtime_input/` 目录
- 支持 `--reset-output` 在启动前清空旧数据
- 随机种子固定为 42（`--seed`），保证结果可复现

### 3.3 双模式支持

| 特性 | 文件模式（默认） | Kafka 模式 |
|------|-----------------|-----------|
| 参数 | `--mode files` | `--mode kafka` |
| 输出目标 | `realtime_input/batch_XXX.json` | Kafka topic `ecommerce-transactions` |
| 依赖 | 无外部依赖 | `kafka-python` 库 + Kafka Broker |
| 适用场景 | 单机演示、教学 | 分布式、生产级 |

## 四、流处理引擎（realtime_processor.py）详细设计

### 4.1 整体架构

`realtime_processor.py` 是基于 **Spark Structured Streaming** 的微批次处理引擎。核心机制是通过 `foreachBatch` 回调函数 `process_batch()`，在每个微批次触发时，对到达的事件集执行一次完整的分析流程。

### 4.2 数据源适配（文件 vs Kafka）

**文件源**（`apps/realtime_processor.py:196-199`）：

```python
parsed_stream_df = (
    spark.readStream.schema(build_schema()).json(args.input_dir)
)
```

直接监听输入目录，当 producer 写入新的 JSON 文件时自动发现并读取。

**Kafka 源**（第 183-194 行）：

```python
raw_stream_df = (
    spark.readStream.format("kafka")
    .option("kafka.bootstrap.servers", args.bootstrap_servers)
    .option("subscribe", args.topic)
    .option("startingOffsets", args.starting_offsets)
    .option("failOnDataLoss", "false")
    .load()
)
parsed_stream_df = raw_stream_df.selectExpr("CAST(value AS STRING) AS json_value")
parsed_stream_df = parsed_stream_df.select(
    from_json(col("json_value"), build_schema()).alias("event")
).select("event.*")
```

Kafka 模式下增加了 JSON 解析环节，通过 `from_json()` 将 Kafka 消息反序列化为结构化 DataFrame。配置 `failOnDataLoss=false` 保证在 Kafka 日志被清理时不会导致查询失败。

### 4.3 统一数据 Schema

两种模式共用同一套 Schema 定义（`build_schema()` 函数，第 102-119 行），共 13 个字段：

```python
StructType([
    StructField("event_id",       StringType(),  False),  # 事件唯一 ID
    StructField("event_time",     StringType(),  False),  # 事件发生时间
    StructField("customer_id",    IntegerType(), False),  # 客户 ID
    StructField("gender",         StringType(),  True),   # 性别
    StructField("age",            IntegerType(), True),   # 年龄
    StructField("city",           StringType(),  True),   # 城市
    StructField("membership_type",StringType(),  True),   # 会员类型
    StructField("total_spend",    DoubleType(),  True),   # 消费金额
    StructField("items_purchased",IntegerType(), True),   # 购买件数
    StructField("average_rating", DoubleType(),  True),   # 平均评分
    StructField("discount_applied",BooleanType(),True),   # 是否使用折扣
    StructField("days_since_last_purchase",IntegerType(),True),  # 距上次购买天数
    StructField("satisfaction_level",StringType(),True),  # 满意度等级
])
```

### 4.4 数据增强（Event Enrichment）

`enrich_stream()` 函数（第 148-165 行）在原始事件上增加两个派生字段：

**风险客户标记 `is_at_risk`**：

```python
is_at_risk = when(
    (col("days_since_last_purchase") > 30)
    & (
        (col("satisfaction_level") == lit("Unsatisfied"))
        | (col("average_rating") < lit(3.8))
    ),
    lit(True),
).otherwise(lit(False))
```

判定逻辑：距上次购买超过 30 天，且（不满意 或 评分低于 3.8）。这一定义综合考虑了"时间维度"和"情感维度"两个退货/流失风险指标。

**高价值订单标记 `high_value_order`**：

```python
high_value_order = when(col("total_spend") >= 1000.0, lit(1)).otherwise(lit(0))
```

单笔消费 ≥ 1000 元标记为高价值订单。

### 4.5 批次处理核心逻辑

`process_batch(batch_df, batch_id)` 是流处理的真正入口（第 202-308 行），每个微批次触发时执行以下流程：

#### Step 1: 空批次过滤

```python
event_count = batch_df.count()
if event_count == 0:
    return
```
跳过空批次，避免无事件时写入空文件。

#### Step 2: 事件留存（Event Persistence）

```python
ordered_df = batch_df.orderBy("event_time", "event_id")
latest_events_pdf = ordered_df.toPandas()

# 按批次独立保存
write_csv(latest_events_pdf, f"events/batch_{int(batch_id):03d}.csv")
# 覆盖写入最新批次快照
write_csv(latest_events_pdf, "latest_events.csv")
# 追加写入全量事件历史
append_csv(latest_events_pdf, "stream_events.csv")
```

三种写入策略各有用途：
- `batch_XXX.csv`：批次级归档，支持逐批追溯
- `latest_events.csv`：最新批次快照，供仪表盘调用
- `stream_events.csv`：累计事件日志，保留整个会话的完整事件记录

#### Step 3: 城市维度聚合（City Summary）

```python
city_summary_pdf = (
    ordered_df.groupBy("city")
    .agg(
        count("*").alias("event_count"),
        spark_round(avg("total_spend"), 2).alias("avg_total_spend"),
        spark_sum("items_purchased").alias("total_items_purchased"),
    )
    .orderBy("city")
    .toPandas()
)
write_csv(city_summary_pdf, "city_summary_latest.csv")
```

输出字段：各城市的**交易事件数**、**平均消费金额**、**购买总件数**。每次批次覆盖更新，反映最新批次的各城市表现。

#### Step 4: 会员维度聚合（Membership Summary）

```python
membership_summary_pdf = (
    ordered_df.groupBy("membership_type")
    .agg(
        count("*").alias("event_count"),
        spark_round(avg("total_spend"), 2).alias("avg_total_spend"),
        spark_round(avg("average_rating"), 2).alias("avg_rating"),
        spark_sum("high_value_order").alias("high_value_orders"),
    )
    .orderBy("membership_type")
    .toPandas()
)
write_csv(membership_summary_pdf, "membership_summary_latest.csv")
```

按 Gold / Silver / Bronze 会员等级分组，输出各等级的**事件数**、**平均消费**、**平均评分**、**高价值订单数**，支持不同会员等级的行为对比。

#### Step 5: 风险客户识别（At-Risk Customer Detection）

```python
risk_pdf = (
    ordered_df.filter(col("is_at_risk"))
    .select(
        "event_time", "customer_id", "city", "membership_type",
        "total_spend", "average_rating",
        "days_since_last_purchase", "satisfaction_level",
    )
    .orderBy(col("days_since_last_purchase").desc(), col("average_rating").asc())
    .toPandas()
)
write_csv(risk_pdf, "at_risk_customers_latest.csv")
```

筛选 `is_at_risk=True` 的客户，按**风险程度排序**（距上次购买天数降序、评分升序 = 风险最高的排在最前），输出 8 个关键字段供业务决策参考。

#### Step 6: KPI 指标汇总

```python
metrics_row = pd.DataFrame([{
    "processed_at":           datetime.now().isoformat(timespec="seconds"),
    "batch_id":               int(batch_id),
    "processed_batches":      state["processed_batches"],
    "event_count":            event_count,
    "cumulative_event_count": state["cumulative_event_count"],
    "total_revenue":          round(float(latest_events_pdf["total_spend"].sum()), 2),
    "avg_spend":              round(float(latest_events_pdf["total_spend"].mean()), 2),
    "avg_rating":             round(float(latest_events_pdf["average_rating"].mean()), 2),
    "discounted_orders":      int(latest_events_pdf["discount_applied"].astype(bool).sum()),
    "high_value_orders":      int(latest_events_pdf["high_value_order"].sum()),
    "at_risk_count":          int(len(risk_pdf)),
}])
append_csv(metrics_row, "kpi_history.csv")
write_csv(metrics_row, "latest_batch_metrics.csv")
```

每条 KPI 记录包含 11 个核心指标，形成**批次级 KPI 时间序列**（`kpi_history.csv` 追加）和**最新批次快照**（`latest_batch_metrics.csv` 覆盖）。

#### Step 7: 仪表盘渲染

每批次处理完毕后调用 `render_dashboard(args.output_dir)` 刷新 HTML 仪表盘和 Markdown 摘要。

### 4.6 流控制与停止策略

处理器采用双重停止条件（第 307-330 行）：

| 条件 | 参数 | 说明 |
|------|------|------|
| 批次上限 | `--max-batches`（默认 5） | 处理足够批次后自动停止，适配演示场景 |
| 空闲超时 | `--idle-timeout-seconds`（默认 12 秒） | 长时间无新事件到达时自动停止，避免挂起 |

微批次触发间隔通过 `--trigger-seconds`（默认 5 秒）控制，确保在 producer 写入下一批事件后及时触发处理。

状态通过 `state` 字典跟踪：

```python
state = {
    "processed_batches": 0,
    "cumulative_event_count": 0,
    "should_stop": False,
    "last_batch_completed_at": None,
}
```

### 4.7 安全机制

`safe_reset_output()` 函数（第 122-132 行）在 `--reset-output` 模式下包含路径安全检查：

```python
def safe_reset_output(output_dir, checkpoint_dir):
    realtime_root = os.path.abspath(output_dir)
    if os.path.basename(realtime_root) != "realtime":
        raise ValueError(f"Refusing to delete unexpected output directory: {output_dir}")
    if os.path.exists(realtime_root):
        shutil.rmtree(realtime_root)
```

通过校验 `basename == "realtime"` 防止误删其他目录，提高了操作的健壮性。

## 七、Kafka 模式说明

除默认文件流模式外，系统还预留了完整的 Kafka 生产-消费链路：

### 7.1 原始参考实现

`Main/For Streaming-data/` 目录下的两个脚本提供了 Kafka 链路的原始参考实现：

- **kafka_producer_ecommerce.py**：纯随机数据生成器，内部生成 `customer_id`、`gender`、`age`、`city` 等字段，以 CSV 字符串格式发送到 Kafka，每批 10 条，间隔 20 秒
- **kafka_consumer_ecommerce_to_pg.py**：基于 PySpark Structured Streaming 的 Kafka 消费者，使用 `from_csv()` 解析 CSV 格式消息，按会员类型+年龄组+城市进行客户分群聚合，并通过 JDBC 写入 PostgreSQL 数据库 (`ecommerce_db`)

### 7.2 本项目适配版

本项目在此基础上进行了重要改进：

1. **数据质量提升**：用种子变异替代纯随机生成，保证事件数据的逼真度和业务语义一致性
2. **双模统一**：producer 支持 `--mode files` / `--mode kafka` 切换，processor 支持 `--source-type files` / `--source-type kafka` 切换，同一套代码适配不同运行环境
3. **文件流优先**：教学演示场景下无需额外启动 Kafka/Zookeeper，降低环境复杂度
4. **JSON 格式统一**：替代原始参考中的 CSV 字符串格式，使用结构化 JSON 传递事件，Schema 更清晰，扩展性更好

### 7.3 Docker 编排支持

`docker-compose.yml` 中通过 `profiles: [kafka]` 按需启用 Zookeeper + Kafka Broker：

```yaml
kafka-broker:
  image: confluentinc/cp-kafka:5.4.0
  profiles:
    - kafka
  environment:
    KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka-broker:29092,PLAINTEXT_HOST://localhost:9092
```

默认 `docker-compose up` 不启动 Kafka，只有显式指定 `--profile kafka` 时才启用，保持教学演示的简洁性。

## 九、与批处理链路的对比与互补

| 维度 | 批处理分析 | 实时处理分析 |
|------|-----------|-------------|
| 数据范围 | 全量历史数据（约 35 万条） | 最新交易事件流 |
| 处理方式 | 一次性全量计算 | 微批次增量更新 |
| 分析重点 | 用户分群、城市消费统计、满意度预测 | 实时 KPI、风险客户动态监测 |
| 时效性 | 静态离线 | 持续动态 |
| 技术栈 | PySpark DataFrame API | Spark Structured Streaming |
| 输出 | HTML 分析报告 + 图片 | 实时仪表盘 + CSV 时间序列 |

两条链路互补，使系统能够同时回答"过去发生了什么"和"现在正在发生什么"两类问题。

## 十、关键技术要点总结

1. **种子变异而非纯随机**：`mutate_row()` 以原始数据为种子施加受控扰动，保证生成事件的统计逼真度和业务语义一致性
2. **双模式统一架构**：producer 和 processor 均通过参数切换文件/Kafka 模式，同一份代码适配不同部署场景
3. **foreachBatch 微批次处理**：每个微批次独立触发完整的分析流程（事件留存 → 多维聚合 → 风险识别 → KPI 汇总 → 仪表盘刷新），处理逻辑在 `process_batch()` 中集中实现
4. **写入策略分离**：快照类输出用 `write_csv` 覆盖、历史类输出用 `append_csv` 追加，避免大量小文件的碎片化问题
5. **仪表盘无外部依赖**：HTML 仪表盘采用纯内联 CSS + 原生 HTML 表格/柱状图，无需任何前端框架，直接浏览器打开即可查看
6. **安全的路径操作**：`safe_reset_output()` 通过 basename 校验防止误删
7. **双重停止策略**：批次上限 + 空闲超时，适配无人值守的演示场景
8. **Docker 容器化集成**：通过 `docker-compose.yml` 中的卷挂载（`./apps:/opt/spark-apps`，`./Reports:/opt/spark-results`）将实时处理脚本和输出与宿主机打通，实现"代码在本地编辑—容器内运行—结果在本地查看"的开发闭环
